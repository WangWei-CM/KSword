# KSword 独立命令行插件规范（v1）

## 目标与边界

插件是 `plugin\<plugin-id>\` 下的独立程序，不是 DLL，也不会被
Ksword 进程加载。Ksword GUI 只读取清单、验证入口仍在插件目录内，然后用
`QProcess` 直接启动该入口程序。它不会经过 `KswordCLI.exe`，也不会把插件代码、
Python、ONNX Runtime 或模型加载进 Ksword 进程。

只安装可信插件。进程隔离避免插件崩溃或依赖冲突影响 Ksword 本体，但任何
本地可执行插件本质上仍能以当前用户权限运行。

## 目录与发现

```text
plugin/
  file-analysis/
    plugin.json
    scanner.py
    requirements.txt
    Pefile_General_T1.onnx
    features.json
    LICENSE.txt
    NOTICE
```

宿主按以下优先级查找根目录：

1. 环境变量 `KSWORD_PLUGIN_ROOT`；
2. 当前工作目录下的 `plugin\`；
3. `Ksword5.1.exe` 所在目录及其最多六级父目录下的 `plugin\`。

KSword 发行包**不预装第三方插件载荷**：不要把模型、Python 依赖、插件 EXE 或
`plugin\` 目录复制进 `Release\`、Qt 资源、DLL 或驱动。这样插件模型不会增大
KSword 安装包；插件由商城在用户明确同意许可证后按需安装。

首次一键安装时，宿主会优先使用 `KSWORD_PLUGIN_ROOT`；未设置时会在
`Ksword5.1.exe` 同级创建 `plugin\`，并安装为
`plugin\<install_directory>\`。已存在的插件根目录仍按前述发现顺序使用。

## 清单：`plugin.json`

清单必须是 UTF-8、最大 64 KiB 的 JSON 对象。v1 宿主读取并要求以下顶层字段；
额外字段可以存在，供插件自行使用。

| 字段 | 要求 |
| --- | --- |
| `ksword_plugin_api` | 固定为字符串 `"1"`。 |
| `id` | 与目录名相同；仅小写 ASCII 字母、数字和连字符，最多 64 字符。 |
| `name`、`version`、`description` | 用户可见元数据。 |
| `runtime` | `python` 或 `executable`。 |
| `entrypoint` | 插件目录内的相对文件路径；不能是绝对路径、不能含 `.`/`..` 路径段。 |
| `default_command` | 宿主传给入口的默认子命令，不能含路径穿越。 |
| `targets` | 非空字符串数组，只能包含 `file` 和/或 `process`，且不能重复；它决定 GUI 中显示在哪些“插件”菜单。 |

`python` 运行时优先使用 `KSWORD_PLUGIN_PYTHON`，其次查找 `python.exe`，最后
使用 `py.exe -3`。Python 依赖必须随插件以 `requirements.txt` 说明，不能要求
Ksword 本体嵌入 Python。

## 启动协议

Ksword 直接启动清单入口。对 `runtime: "python"` 且 `default_command: "scan"` 的插件，实际命令等价于：

```text
python.exe <entrypoint> --ksword-plugin scan -- <plugin-arguments...>
```

对 `runtime: "executable"`，Ksword 直接运行 `<entrypoint>` 并追加相同参数。Python 运行时依次查找 `KSWORD_PLUGIN_PYTHON`、`python.exe`、`py.exe -3`。

`--` 后面的参数由宿主原样传递，包含插件自己的 `--help`。插件必须把
`--ksword-plugin <command> -- ...` 视为非交互模式：不得等待 stdin，不得显示
GUI，也不得把人类横幅混入 stdout。

## Ksword 图形界面入口

GUI 中插件一律按“**插件 → <插件名>**”显示，名称和可用目标直接来自 `plugin.json`，没有针对某个插件的硬编码菜单。

- 文件管理器：选择单个常规文件后，从 **插件 → <支持 `file` 的插件>** 调用。
- 进程详细信息：打开任一进程的 **插件** 页，从 **插件 → <支持 `process` 的插件>** 调用。
- 顶部功能条：**插件管理** 按钮打开“插件管理”。它可重新扫描、查看只读清单详情、打开插件目录，并从商城一键安装独立插件。

文件目标的 GUI 调用示例：

```text
<entrypoint> --ksword-plugin scan -- --target-kind file --path <selected-file>
```

进程目标使用稳定上下文参数；插件不得把 PID 当作内核地址或命令执行载荷：

```text
<entrypoint> --ksword-plugin <default-command> -- --target-kind process --pid <pid> --path <image-path> --process-name <name>
```

因此 GUI 只通过命令行协议调用插件：Ksword 直接读取本地清单并启动入口程序，不会导入 Python、ONNX Runtime 或任何插件代码。查找顺序为 `KSWORD_PLUGIN_ROOT`、当前工作目录、主程序同目录的发行布局及源码树的开发布局。启动失败时 GUI 会给出清单、入口文件或运行时的明确提示。

## 插件商城

插件管理窗口会异步读取公开目录：

```text
https://raw.githubusercontent.com/KSwordDEV/Plugins/main/catalog.json
```

目录根对象必须含有 `ksword_plugin_marketplace_api: "1"`。每个条目必须包含 `id`、`name`、`version`、`description`、`targets`、`install_directory`、`archive_url`、`sha256`、`license_name` 和 `license_url`。下载和许可证 URL 仅接受 `raw.githubusercontent.com` 的 HTTPS 地址；ZIP 必须有 64 位 SHA-256。

商城一键安装流程不可绕过许可证：先下载并展示 `license_url` 的全文，用户勾选“已阅读并同意”后，才允许请求插件 ZIP。ZIP 下载完成后，Ksword 会校验 SHA-256；校验失败不会解压或写入插件目录。校验通过后，宿主在插件根目录中建立暂存区，用 Windows PowerShell `Expand-Archive` 解压，并再次验证解压后的 `plugin.json`、目录名、ID 和入口文件。验证通过才替换同 ID 的旧版本；失败时保留旧版本并清理暂存内容。安装完成后“已安装”页自动重新扫描。

商城下载的 ZIP 不写入用户的下载目录，也不会随 KSword 安装包分发。许可证同意不会持久化：每次一键安装或更新均需再次阅读并确认。

## 输出、退出码与兼容性

插件模式的 stdout 是 UTF-8 JSON Lines，每一行必须至少包含：

```json
{"protocol":"ksword-plugin/1","plugin_id":"example","event":"..."}
```

建议依次发出 `ready`、`scan_started`、零或多条 `file_result`、`scan_complete`。
失败用 `event: "error"` 并提供稳定的 `code` 和 `message`。stderr 保留给人工
诊断。退出码 `0` 表示命令完成；`64` 表示参数错误；`2` 表示目标或运行环境不
可用；其它非零值由插件自行定义，并由 Ksword 直接显示。

宿主不会解析或信任插件结果来执行 R0 操作。插件若需要改变系统状态，必须自行
提供独立、明确且受其自身安全策略约束的命令。

## 发布与第三方内容

每个第三方插件必须在根目录保留原始 `LICENSE` 和 `NOTICE`，并在清单中引用。
发布包必须携带其模型、特征文件、依赖说明和许可证。第三方插件发布在
`KSwordDEV/Plugins`，不提交进 KSword 主仓库，也不编入 Ksword 安装包。当前
`ARK/PYAS-Scanner` 是参考实现；它保留原始交互式调用，同时实现上述 JSON Lines
插件入口。
