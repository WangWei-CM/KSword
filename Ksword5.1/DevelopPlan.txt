1.日志与管理
	1.1在KswordARK根目录下新建Framework.h，新建Framework文件夹；Framework文件夹中有Log.cpp。Framework.h是整个Framework文件夹对外调用的头文件。
	1.2在log.cpp中创建五个基于cout的对象，名字分别为dbg，info，warn，err，fatal。创建一个宏定义eol（end of log）如下：file(__FILE__)<<line(__LINE__)<<func(__FUNCTION__)...<<std::endl，反正把那个能定位位置的全部加进去，我记得msvc有特殊的宏可以直接把整个函数的声明弄进去来着。
	1.3规定一个event类，该对象仅包含一个声明为const 的GUID，构造时自动随机初始化。每条日志语句调用时必须传入一个event，然后再像正常cout那样调用。
	1.4调用示例代码如下：
	{event tmp;info<<tmp<<"现在输出一些文字"<<eol}
	{event tmp2;err<<tmp<<"现在输出一些文字"<<eol}
	在控制台产生如下输出：
	[ + ][2026-04-23 11:45:14]现在输出一些文字
	[  x ][2026-04-23 11:45:15]现在输出一些文字(File:xxx.cpp, Line 14, int function(int)))
	仅err和fatal输出调用位置。
	1.5外部调用时（只考虑Windows平台），彩色打印日期和时间戳过后，用白色像cout那样正常输出后面的东西。每一个类的颜色与图表分别为：dbg 蓝色 [ (空) ]；info 绿色 [ + ]；warn 黄色 [ ! ]；err 红色 [ x ]；fatal红底白字 [***]
	1.6五个类必须保证线程安全，互相之间也是一样，尤其是控制台输出的时候。
	1.7声明一个类叫做kEvent的类，储存每一次使用流式日志产生的输出。该类包含一个GUID，一个string储存输出的日志内容，一个string储存文件的位置，一个储存函数名字，一个time类型的储存时间戳。每次调用前面写到的五个类的时候就填充一个kEvent，然后添加到日志管理器。日志管理器定义如下
	1.8日志管理器：声明一个kEventEntry类，包含一个std::vector <kEvent>以及如下成员函数：void add(kEvent)，void clear(),，bool Save(std::string)用于把所有列表中的内容储存到string中包含的路径，bool返回值表示是否成功，std::vector <kEvent> Track(GUID)，传入一个GUID返回所有储存记录中同样GUID的事件列表。
	1.9全局声明一个kEventEntry的变量KswordARKEventEntry，实现在log.cpp中，extern在Framework.h中。
	1.10在图形界面中的日志列表中开一个不可编辑表格，占满整个docking widget，不要求最小尺寸，共_列，分别分为等级，时间，内容，文件，函数列，等级列的标题为空，仅放置一个对应颜色的正方形在里面即可，该列列宽锁定为这个正方形。特别地，fatal整行黑底白字，err整行红底黑字。其他列允许调整列宽，每一个单元格都可以右键，菜单中有：复制单元格；复制行；跟踪事件。点击跟踪事件后该列表全部暂时只显示同GUID的事件。右键这个状态下厚的任意单元格，菜单中的第三项改为取消追踪，点击后返回正常界面。
	1.11此表格上方添加五个复选框，分别是从Debug到Fatal的所有等级默认全部勾选，列表中只展示对应的等级。
	1.12表格上方再增加一拍按钮，如下:导出日志、清空日志列表（二级确认提示框）。其中导出日志选项点击后弹出shell菜单选择目标路径，保存的文件后缀为.txt。如果打开shell失败意味着explorer.exe已经崩溃，此时弹出一个提示框要求输入路径。把整个日志系统中保存的所有项（包括GUID）全部保存，用\t分隔项，用换行分隔行。增加“复制可见”，把现在列表中的内容进行复制，主要是适应包含追踪列表的那部分。增加复选框，保持表格滚动到最底端。
	1.13把上述中的日志系统调用方法和语法规范写进开发规范.md里面放在根目录下。
2.进度条管理
	2.1新增一个kProgress类，这个类负责管理进度条的显示。创建一个此类的全局对象命名为kPro，应该有如下功能：
	int kPro.add("任务1", "步骤1")返回一个int，作用相当于pid，和任务绑定
	void kPro.set(PID,"步骤2" ，0, 70f)设置目标pid的任务为步骤2，进度条70%。特别地，如果设置为1.00f，那么把这个kPro在列表中隐藏
	int kPro.UI(PID，"请选择你的操作"，"选项A"，“选项B”...)可变参阻塞式，返回int从1开始对应选择了传入参数的第几个按钮选项。
	2.2 当调用add的时候，任务列表增加一张卡片，包含任务与步骤和一个进度条，调用set的时候把指定的PID进度条和步骤改为传参中写的那样。调用UI的时候出现按钮选择，进度条小时，完成选择后进度条重新出现。
3.进程
	3.1在写有进程的Docking Widget中添加侧边栏Tab栏，里面写上“进程列表“。
	3.2和操作系统的交互封装的函数放在根目录下Ksword文件夹中。这个文件夹提供一个放在根目录下的Ksword.h文件，和随着命名空间同步嵌套的文件夹递归。例如你可能写到的GetProcessNameByPID()，他应该放在ksword/process/process.cpp中，访问这个函数需要ks::process::GetProcessNameByPID()。ks小写，其他你自己规定，需要进行字符串转换的也是工具函数，也塞进ksword文件夹去。核心是脱离这个kswordARK项目后这个文件夹和Ksword.h头文件可以给其他程序提供快速的win32API
	3.3 每一个命名空间对应的文件夹下都有一个头文件，一级一级向上包含。例如在上一个例子里面，要有process.h。最终framework.h包含ksword.h即可完成所有工具函数的引用。所有项目文件都要包含framework.h。以上内容写入代码开发规范那个md文件。
	3.4 在这个进程列表tab中完成一个r3的任务管理器。最上方有下拉菜单指定进程遍历方案，包括创建那个什么snapshot，processnext和使用NtQuerySystemInformation。
	3.5任务管理器上方有一个按钮，切换使用树状图或者列表形式显示。树状图也以列表的形式呈现，以父子进程的顺序给进程名的那一列添加缩进和标尺。
	3.6 表格应有进程名（实际显示时前面渲染进程logo）列、PID、CPU、RAM、DISK、GPU、DIsk、Net、数字签名、路径、父进程、启动参数、用户、启动时间、是否管理员列。在表格上方增加切换视图模式，提供两种视图，第一种是监视视图，只显示进程名，pid和性能计数器那些列，第二种是详细信息视图，显示除此之外的其他信息。用户也可以右键标题栏勾选/取消勾选显示哪些列，也可以拖动排序、调整列宽。默认视图模式是性能计数器
	3.7增加开始监视、暂停监视按钮;增加滑块设置几秒钟刷新一次。进程列表获取以及详细信息获取必须做成异步的，不要阻塞图形界面。
	3.8对于新的一次刷新进程列表，如果PID和进程创建时间都是相同的，则认为这个进程之前有过，复用上次的数据而不重新获取（性能计数器的数据除外）
	3.9对于新的一次刷新进程列表，上一次后新增的进程用高亮绿色作为背景，上一次后推出的进程不刷新数据，但是这一轮保留显示，背景做成灰色
	3.10每个进程都可以右键。单元格右键菜单如下：复制单元格，复制行，结束进程（二级菜单：Taskkill，Taskkill /f，TerminateProcess，TerminateThread（结束每一个这个进程的线程），注入无效shellcode几种选项），挂起进程，恢复进程，设为关键进程，取消关键进程（如果无法实现请告诉我，我会在下一个回答提供核心代码）、打开所在目录、设置进程优先级（提供具体的二级菜单）、进程详细信息。详细信息暂时留空。
